# v0.5 Enable-First Design

## Goal

Solve two handoff problems without writing private agent memory into the
business repository:

- A new session on the same device should recover previous project context and
  user preferences.
- A session on another clone, worktree, directory, or device should be able to
  recover the same memory when sync is configured.

## Product Model

`agent-handoff` is a user-level memory vault plus a user-level skill.

Default local state:

```text
~/.agent-handoff/
  config.json
  vault/
    global/
      preferences.md
      lessons.md
    projects/
      github.com__owner__repo/
        project.md
        decisions.md
        preferences.md
        branches/
          main.md
        checkpoints/
```

The CLI never modifies repository instruction files such as `AGENTS.md` or
`CLAUDE.md`. Normal repositories do not need `.agent-handoff.yml`; project
identity is derived from `git remote get-url origin`. `.agent-handoff.yml` is
only an optional override for unusual layouts.

## Commands

```bash
agent-handoff enable
agent-handoff start
agent-handoff checkpoint --note "..."
agent-handoff learn --kind preference --note "..."
agent-handoff sync init <private-git-url>
agent-handoff sync
agent-handoff status
```

`enable` creates local memory and installs the `agent-handoff` skill. Cross
device sync is intentionally separate: users who only need same-device session
handoff do not need a git vault repository.

`status` reports both readiness and whether sync is configured so an agent can
decide whether to run `sync` before `start`.

## Agent Flow

At session start, the installed skill tells the agent to run `status`, then
`start` once enabled. Before pausing, the agent writes a checkpoint. Stable user
preferences, recurring corrections, durable project decisions, and branch
context go through `learn`.

The tool is deliberately file-backed and CLI-driven for the MVP. There is no
daemon, MCP server, prompt injection into user repos, or automatic background
sync.
